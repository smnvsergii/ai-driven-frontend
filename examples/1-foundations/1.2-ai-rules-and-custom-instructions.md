# 1.2 AI Rules / Custom Instructions

**🌐 Language:** **English** · [Українська](./1.2-ai-rules-and-custom-instructions.ua.md)

**Block:** Foundations
**Practice project:** [`phaser-memory-cards-MFE`](https://github.com/smnvsergii/phaser-memory-cards-MFE) — Phaser 3 + Vite + TypeScript memory game that also runs as a microfrontend.

> Goal of this note: instead of repeating my project's conventions in every prompt (block 1.1), I encode them **once** into a rules file the agent reads automatically. Below is the real file I added to the game and the reasoning behind every section.

---

## The problem this solves

In [1.1](./1.1-prompt-and-context-engineering.md) I learned to pass context per prompt. But the *same* context — "this is a canvas not a DOM app", "4-space indent", "config lives in `config.ts`" — was needed in almost every prompt. Repeating it is wasteful and easy to forget, and a forgotten constraint means a wrong answer.

**AI rules files** move those stable constraints out of the prompt and into the repo, where the agent loads them automatically on every request.

## What I added to `memory-cards`

A single [`AGENTS.md`](https://github.com/smnvsergii/phaser-memory-cards-MFE/blob/main/AGENTS.md) at the repo root.

### Why `AGENTS.md` and not `.cursorrules`

There are several formats: `.cursorrules` (Cursor), `.github/copilot-instructions.md` (Copilot), `CLAUDE.md` (Claude Code), and the tool-agnostic `AGENTS.md`. I picked **one canonical `AGENTS.md`** for two reasons:

1. **Avoid drift.** Multiple rules files inevitably fall out of sync. One source of truth is the whole point.
2. **Portability.** `AGENTS.md` is an emerging open convention that most agents read; if a specific tool needs its own filename, I'd make it a thin pointer to `AGENTS.md` rather than duplicate the content.

> 💡 Takeaway: treat AI rules like any shared config — one source of truth, no copies that can drift.

## How AI tools actually find it

A fair question: *if there's one file, how does each tool see it?* In 2024–2025 every tool had its own filename (`.cursorrules`, `.github/copilot-instructions.md`, `CLAUDE.md`, …). By 2026 the industry converged on **`AGENTS.md` at the repo root** as an open standard (stewarded by the Linux Foundation's Agentic AI Foundation). Most agents — Cursor, GitHub Copilot, Codex, Gemini CLI, Windsurf, Aider, Zed and 20+ others — now read it **natively** when a session starts.

A few tools still expect their own filename. For those I add a **thin shim** — a one-line file that points back to `AGENTS.md`, never a copy:

| Tool | File in repo | Role |
| ---- | ------------ | ---- |
| Most agents | `AGENTS.md` | Source of truth |
| Claude Code | `CLAUDE.md` | Shim → `@AGENTS.md` import |
| Kiro | `.kiro/steering/project-rules.md` | Shim → includes `AGENTS.md` |

Example shim (`CLAUDE.md`):

```markdown
See @AGENTS.md for all project instructions.
```

This keeps the "one source of truth" property while still covering tools that don't read `AGENTS.md` natively. The mechanics are also documented inside `AGENTS.md` itself under *"How tools discover this file"*, so anyone opening the repo understands the setup.

> 💡 Takeaway: shims point, they don't copy. One file holds the rules; every other filename is a redirect to it.

## How the file is structured

Each section exists to kill a class of mistakes I actually saw the model make.

| Section | The mistake it prevents |
|---------|--------------------------|
| **Project in one line** | Agent treating a Phaser canvas game like a React/DOM app. |
| **Tech stack (don't introduce alternatives)** | Pulling in React, a CSS framework, or a state library "to help". |
| **Commands** | Inventing script names instead of using the real `npm run` scripts. |
| **Before you finish (verification)** | Declaring "done" without running typecheck / lint / build. |
| **Code style** | Wrong indentation/quotes that the pre-commit hook would reject anyway. |
| **Architecture & conventions** | Loading assets in the wrong scene, hardcoding magic numbers, using CSS media queries for canvas layout. |
| **MFE protocol** | Breaking the `MFEEnvelope` discriminated union or silently desyncing from the shell. |
| **Guardrails** | Editing generated dirs, committing unprompted, refactoring unrelated code. |

### Principles behind good rules

- **Make rules verifiable, not vibes.** "Run `npm run typecheck`, `lint`, `build` before finishing" beats "write good code".
- **Tie rules to real artifacts.** Reference `config.ts`, `protocol.ts`, the actual Prettier settings — so the agent can check itself.
- **State invariants explicitly.** Example: *the flip animates `scaleX` only, never `scaleY`*. That's a non-obvious rule a fresh agent would break.
- **Keep it skimmable.** Short sections, tables, imperative voice. Rules nobody reads (human or model) are dead weight.
- **Encode the "don'ts".** A guardrails section ("don't add deps for things Phaser already does") prevents the most expensive mistakes.

## Before / after

**Before (per-prompt, from 1.1):** every request had to restate the stack, the style, and "this is a canvas".

**After:** the prompt shrinks to the actual task —

```
Add a "hintUsed" event to the MFE protocol with payload { remaining: number }.
```

— and the agent already knows to follow the `MFEEnvelope` pattern, keep `protocol.ts` in sync, respect strict TS, and run verification, because `AGENTS.md` told it once.

## Result

The rules file turned the lessons from 1.1 into something durable. Prompts got shorter and answers got more consistent, because the constraints that used to live in my head (and get forgotten) now live in the repo and load automatically.
