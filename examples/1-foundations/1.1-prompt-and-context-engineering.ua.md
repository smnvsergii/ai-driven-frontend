**🌐 Мова:** [English](./1.1-prompt-and-context-engineering.md) · **Українська**

**Блок:** Foundations

## 1. Структура промпта

Слабкий промпт описує бажання. Сильний промпт описує **роль, мету, жорсткі обмеження та формат відповіді**.

### Антипатерн (розпливчасто)

```
зроби картки адаптивними
```

Моделі доводиться вгадувати фреймворк, правила розкладки, брейкпоінти й те, що тут означає «адаптивно». Зазвичай вона вигадує CSS media queries — що неправильно, бо це Phaser-канвас, а не DOM-розкладка.

### Патерн (структуровано)

```
Роль: Ти працюєш у грі на Phaser 3 + TypeScript (canvas, не DOM).

Мета: Розкласти 8 карток у сітку, що перебудовується при зміні розміру вікна.

Обмеження:
- Scale manager працює в режимі RESIZE; перерозкладка — на події сцени 'resize'.
- Завжди 4 колонки (див. GameConfig.grid.breakpoints); картки зменшуються, а не переходять у менше число колонок.
- Картки не повинні перекривати лого, вшите у фон
  (резервуй GameConfig.layout.logoBottomRatio від висоти фонового зображення).
- Дотримуйся cardAspect (2/3) і maxCardHeight (300).

Формат: одна функція розкладки, що повертає {x, y, width, height} для кожного індексу картки.
Не додавай CSS media queries — це канвас.
```

**Чому це працює:** кожне обмеження відповідає реальному значенню в [`src/config.ts`](https://github.com/smnvsergii/phaser-memory-cards-MFE/blob/main/src/config.ts). Модель перестає вгадувати й починає рахувати під мою реальну дизайн-систему.

> 💡 Висновок: обмеження — найважливіша частина промпта з найбільшим важелем. Кожне обмеження відсікає цілий клас неправильних відповідей.

## 2. Передача контексту

Модель настільки хороша, наскільки хороший контекст, який ти їй даєш. Навичка — обрати *мінімально достатній* зріз.

### Приклад: розширення MFE-протоколу

Задача: додати подію `hintUsed` від гри до shell.

❌ **Замало контексту** — «додай подію hintUsed у протокол» → модель вигадує довільну форму повідомлення, що не збігається з наявним патерном envelope.

❌ **Забагато контексту** — вставляємо все дерево `src/` → модель тоне, латентність зростає, вона редагує сторонні файли.

✅ **Правильний контекст** — вставляємо лише [`src/mfe/protocol.ts`](https://github.com/smnvsergii/phaser-memory-cards-MFE/blob/main/src/mfe/protocol.ts) і кажемо:

```
Ось wire-протокол. Події використовують discriminated union MFEEnvelope<Type, Payload>.
Додай подію 'hintUsed' з payload { remaining: number }, точно за патерном
наявної події 'match'. Хелпери MFEEventType / EventPayload мають продовжувати працювати.
```

Тепер модель коректно розширює discriminated union, і хелпер-типи й далі виводяться, бо вона *побачила патерн, який мала наслідувати*.

### Перевикористовуване правило: що включати

1. Файл, який змінюємо.
2. Контракт, який він має поважати (типи, конфіг, інтерфейс) — навіть якщо він в іншому місці.
3. Один наявний приклад патерну для наслідування.
4. Більше нічого.

> 💡 Висновок: інженерія контексту — це переважно *віднімання*. Підбери найменший набір файлів, за якого правильна відповідь стає очевидною.

## 3. Ітеративне уточнення

Перша відповідь — це чернетка. Уточнюй **хірургічними follow-up'ами**, що посилаються на власний вивід моделі, замість того щоб перепитувати з нуля.

### Реальний кейс: анімація перевороту картки

Переворот реалізовано в [`src/objects/Card.ts`](https://github.com/smnvsergii/phaser-memory-cards-MFE/blob/main/src/objects/Card.ts). Вимога: картка має перевертатися через горизонтальне стиснення (scale X → 0, заміна текстури, масштаб назад) **без жодної зміни висоти**.

Наївна перша відповідь масштабувала і X, і Y, тож картки помітно «стрибали» в розмірі посеред перевороту на адаптивній сітці.

Послідовність уточнень, що це виправила:

1. *«Анімація також змінює висоту — картки стрибають на вузьких вікнах. Анімуй лише scaleX; ніколи не чіпай scaleY.»*
2. *«Після заміни текстури нове зображення має іншу нативну ширину, тож старий scaleX неправильний. Перерахуй цільовий scaleX як cardWidth / нативна ширина текстури перед tween-розширенням.»*
3. *«Підтверди, що висота лишається сталою при заміні текстури — проведи мене через математику масштабування.»*

Кожен крок прив'язаний до конкретного симптому й попереднього виводу. Це збіглося за три ходи; цикл «переробимо все з нуля» так не зробив би.

> 💡 Висновок: не починай заново — налагоджуй розмову. Назви симптом, вкажи на конкретний рядок/поведінку, давай одне коригувальне обмеження за раз.

## Результат

Застосування цього до `memory-cards` перетворило «AI вгадує мою архітектуру» на «AI рахує під мою архітектуру». Найбільшим важелем виявилися **обмеження, прив'язані до реальних значень конфігу** — це прибрало майже весь зайвий пінг-понг.
