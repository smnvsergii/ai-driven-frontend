# 1.1 Prompt & Context Engineering

**🌐 Language:** **English** · [Українська](./1.1-prompt-and-context-engineering.ua.md)

**Block:** Foundations
**Practice project:** [`phaser-memory-cards-MFE`](https://github.com/smnvsergii/phaser-memory-cards-MFE) — a Phaser 3 + Vite + TypeScript memory game that also runs as a microfrontend inside a Game Hub shell.

> Goal of this note: show *how* I structure prompts, feed context, and refine iteratively — using real tasks from the game, not toy examples.

---

## What this block covers

| Skill | One-line definition |
|-------|---------------------|
| Prompt structure | Giving the model a role, a goal, constraints, and the expected output format instead of a vague request. |
| Passing context | Deciding *what* slice of the codebase the model needs to see, and feeding exactly that — no more, no less. |
| Iterative refinement | Treating the first answer as a draft and steering it with targeted follow-ups instead of restarting. |

---

## 1. Prompt structure

A weak prompt describes a wish. A strong prompt describes a **role, a goal, hard constraints, and the output format**.

### Anti-pattern (vague)

```
make the cards responsive
```

The model has to guess the framework, the layout rules, the breakpoints, and what "responsive" means here. It usually invents CSS media queries — which is wrong, because this is a Phaser canvas, not a DOM layout.

### Pattern (structured)

```
Role: You are working in a Phaser 3 + TypeScript game (canvas, not DOM).

Goal: Lay out 8 cards in a grid that re-flows on viewport resize.

Constraints:
- Scale manager runs in RESIZE mode; relayout happens on the scene 'resize' event.
- Always 4 columns (see GameConfig.grid.breakpoints); cards shrink, never reflow to fewer columns.
- Cards must never overlap the logo baked into the background
  (reserve GameConfig.layout.logoBottomRatio of the bg image height).
- Respect cardAspect (2/3) and maxCardHeight (300).

Output: a single layout function that returns {x, y, width, height} per card index.
Do not add CSS media queries — this is a canvas.
```

**Why it works:** every constraint maps to a real value in [`src/config.ts`](https://github.com/smnvsergii/phaser-memory-cards-MFE/blob/main/src/config.ts). The model stops guessing and starts computing against my actual design system.

> 💡 Takeaway: constraints are the highest-leverage part of a prompt. Each constraint you add removes a whole class of wrong answers.

## 2. Passing context

The model is only as good as the context window you give it. The skill is choosing the *minimal sufficient* slice.

### Example: extending the MFE protocol

Task: add a `hintUsed` event from the game to the shell.

❌ **Too little context** — "add a hintUsed event to the protocol" → the model invents an ad-hoc message shape that doesn't match the existing envelope pattern.

❌ **Too much context** — pasting the entire `src/` tree → the model drowns, latency spikes, and it edits unrelated files.

✅ **Right context** — paste only [`src/mfe/protocol.ts`](https://github.com/smnvsergii/phaser-memory-cards-MFE/blob/main/src/mfe/protocol.ts) and say:

```
Here is the wire protocol. Events use the MFEEnvelope<Type, Payload> discriminated union.
Add a 'hintUsed' event with payload { remaining: number }, following the exact pattern
of the existing 'match' event. Keep the MFEEventType / EventPayload helpers working.
```

The model now extends the discriminated union correctly and the helper types keep inferring, because it *saw the pattern it had to imitate*.

### A reusable rule for what to include

1. The file being changed.
2. The contract it must honor (types, config, an interface) — even if it lives elsewhere.
3. One existing example of the pattern to imitate.
4. Nothing else.

> 💡 Takeaway: context engineering is mostly *subtraction*. Curate the smallest set of files that makes the correct answer the obvious one.

## 3. Iterative refinement

The first answer is a draft. Refine with **surgical follow-ups** that reference the model's own output, instead of re-prompting from scratch.

### Real case: the card flip animation

The flip is implemented in [`src/objects/Card.ts`](https://github.com/smnvsergii/phaser-memory-cards-MFE/blob/main/src/objects/Card.ts). The requirement: the card must flip by collapsing horizontally (scale X → 0, swap texture, scale back) **without ever changing its height**.

A naive first answer scaled both X and Y, so cards visibly "jumped" in size mid-flip on a responsive grid.

Refinement sequence that fixed it:

1. *"The animation also changes height — cards jump on narrow viewports. Animate scaleX only; never touch scaleY."*
2. *"After swapping the texture, the new image has a different native width, so the old scaleX is wrong. Recompute the target scaleX as cardWidth / texture native width before the expand tween."*
3. *"Confirm height stays constant across a texture swap — walk me through the scale math."*

Each step is anchored to a concrete symptom and the previous output. That converged in three turns; a "redo the whole thing" loop would not have.

> 💡 Takeaway: don't restart — debug the conversation. Name the symptom, point at the exact line/behavior, give one corrective constraint at a time.

---

## Checklist for myself

- [ ] Did I give a role + goal + constraints + output format?
- [ ] Does every constraint map to a real value/contract in the codebase?
- [ ] Did I include the file, its contract, and one example to imitate — and nothing else?
- [ ] When the answer was off, did I refine surgically instead of re-prompting?

## Result

Applying this to `memory-cards` turned "AI guesses my architecture" into "AI computes against my architecture". The biggest single lever was **constraints tied to real config values** — it removed almost all of the back-and-forth.
