# 1.3 Custom Skills / Commands

**🌐 Language:** **English** · [Українська](./1.3-custom-skills-and-commands.ua.md)

**Block:** Foundations
**Practice project:** [`phaser-memory-cards-MFE`](https://github.com/smnvsergii/phaser-memory-cards-MFE) — Phaser 3 + Vite + TypeScript memory game that also runs as a microfrontend.

> Goal of this note: rules (1.2) describe *how* to behave on every task. **Custom commands** package a specific *recurring task* into one invocation, so a multi-step, error-prone procedure becomes `/command`.

---

## Rules vs commands — what's the difference

| | AI Rules (1.2) | Custom Commands (1.3) |
|--|----------------|------------------------|
| Scope | Apply to **every** request | Invoked for **one specific** task |
| Form | Passive context (`AGENTS.md`) | Active, named action (`/add-mfe-message`) |
| Analogy | A project's coding standard | A project's npm script / Makefile target |

Rules stop the agent doing the wrong thing in general. Commands encode the exact steps of a repetitive job so you don't re-explain them every time.

## The recurring task I automated

In `memory-cards`, adding a message to the MFE protocol is a small but **multi-file, easy-to-break** procedure. Adding one command touches:

1. `src/mfe/protocol.ts` — extend the `ShellCommand` / `MFEEvent` discriminated union.
2. `src/mfe/bridge.ts` — add the type to the `SHELL_COMMAND_TYPES` **runtime `Set`**.
3. `README.md` — add a row to the command/event table.
4. The Game Hub shell — mirror the protocol by hand.

The trap: steps 1 and 2 are maintained separately. The TypeScript union is compile-time; the `Set` in `bridge.ts` is runtime. Update the union but forget the `Set`, and the new command **type-checks fine but is silently dropped** at runtime by `isShellCommandType`. I hit this exact bug once — perfect candidate to encode into a command.

## What I added to `memory-cards`

Two Claude Code slash commands in [`.claude/commands/`](https://github.com/smnvsergii/phaser-memory-cards-MFE/tree/main/.claude/commands):

### `/add-mfe-message`

```markdown
---
description: Add a new MFE protocol message (command or event) end-to-end...
argument-hint: <command|event> <messageType> [payloadShape]
allowed-tools: Read, Edit, Bash(npm run typecheck), Bash(npm run lint)
---
```

The body is a numbered checklist of all four steps above, with the runtime-`Set`
trap called out explicitly, plus a final "remind me to mirror it in the shell".
Invocation:

```
/add-mfe-message event hintUsed { remaining: number }
```

### `/verify`

A trivial but high-frequency one: runs `typecheck → lint → build` in order and
reports pass/fail. Saves typing the sequence and guarantees I never skip a step
before calling something "done".

## Anatomy of a good command

- **Frontmatter does the wiring.** `description` (discoverable), `argument-hint`
  (usage), `allowed-tools` (scopes what it can run — `/verify` can only run the
  three npm scripts, nothing else).
- **Arguments make it reusable.** `$ARGUMENTS` / `$1` / `$2` turn a one-off
  prompt into a parameterized tool.
- **Encode the trap, not just the happy path.** The whole value of
  `/add-mfe-message` is that it *remembers the runtime `Set`* — the thing humans forget.
- **Defer to the rules.** Commands say "follow `AGENTS.md`" instead of re-stating
  conventions, so there's still one source of truth (see 1.2).
- **Keep side effects honest.** `/add-mfe-message` refuses to edit the shell repo
  and only *flags* the mirror step — it doesn't pretend to do work it shouldn't.

## Portability note

Slash commands are tool-specific in form: Claude Code uses `.claude/commands/`,
Cursor uses `.cursor/commands/`, etc. Unlike rules (where `AGENTS.md` is a shared
standard — see 1.2), there's no single cross-tool format for commands yet. I keep
the *logic* in the command file and the *conventions* in `AGENTS.md`, so porting
a command to another tool is mostly copy + adjust the frontmatter.

## Checklist for myself

- [ ] Is this a task I do more than a couple of times? (If not, a plain prompt is fine.)
- [ ] Does the command encode the **error-prone** step, not just the obvious ones?
- [ ] Is it parameterized with arguments instead of hardcoded?
- [ ] Does it defer conventions to `AGENTS.md` rather than duplicating them?
- [ ] Is `allowed-tools` scoped to only what the command needs?

## Result

The protocol-message procedure went from "open four files and hope I remember the
runtime `Set`" to one invocation that does it consistently. Commands turned my
most repetitive, bug-prone chore into a single, safe action.
