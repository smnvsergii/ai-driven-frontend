# 2.3 AI Testing

**🌐 Language:** **English** · [Українська](./2.3-ai-testing.ua.md)

**Block:** Daily Workflow
**Practice project:** [`phaser-memory-cards-MFE`](https://github.com/smnvsergii/phaser-memory-cards-MFE) — Phaser 3 + Vite + TypeScript memory game that also runs as a microfrontend.

> Goal of this note: use AI to set up a test stack and generate tests that target
> *real risk* — not coverage theatre. This is the block where the project went from
> zero tests to a working Vitest suite guarding the bug we keep hitting.

---

## Starting point: no tests

Before this block, `memory-cards` had lint, format, typecheck, and build in CI —
but **no test runner at all**. AI testing isn't just "generate tests"; step one is
**choosing and wiring the right stack**, which AI accelerates well.

## What I added to `memory-cards`

### 1. The stack — Vitest + jsdom

Vitest is the natural fit for a Vite project (shared transform pipeline, fast).
The MFE bridge talks to `window` / `postMessage`, so tests need a DOM — hence
**jsdom**. I kept the test config separate from the build:

```ts
// vitest.config.ts
export default defineConfig({
    test: { environment: 'jsdom', include: ['src/**/*.test.ts'], globals: false },
});
```

`globals: false` is a deliberate choice — tests import `describe/it/expect` from
`vitest` explicitly, so there's no ambient magic and the linter stays happy.

### 2. The tests — `src/mfe/bridge.test.ts`

I pointed the tests at the **highest-risk unit**: the `MFEBridge` message layer.
Not the Phaser rendering — canvas output is the wrong thing to unit-test. The
8 tests cover:

- emit produces a well-formed envelope to the parent (embedded mode);
- a registered handler fires for a valid command;
- **every known shell command type is accepted** — this is the test that guards the
  union ↔ runtime `SHELL_COMMAND_TYPES` `Set` desync from [1.3](../1-foundations/1.3-custom-skills-and-commands.md)/[2.2](./2.2-ai-debugging.md);
- an unknown type is dropped;
- messages from a disallowed origin are ignored;
- the `target` id is respected;
- unsubscribe (`off()`) actually stops delivery;
- standalone mode (`window.parent === window`) makes `emit` a no-op.

The recurring bug now has a **failing test if it returns** — the loop from 2.2
("make the bug un-reintroducible") closes here.

### 3. The command — `.claude/commands/write-tests.md`

A `/write-tests <file>` command so generating new tests stays consistent: it
encodes the conventions (file location, explicit `vitest` imports, jsdom, one
behavior per `it`, restore mocks) and tells the AI to **prioritize the risky edges**
and never unit-test Phaser rendering.

### 4. Wiring it into the gate

- `package.json`: `test` (watch) + `test:run` (single run).
- `ci.yml`: a `Test` step between typecheck and build.
- `AGENTS.md`: `test:run` added to the verification sequence, plus a Testing section.

## How I actually used AI here

- **Stack setup:** AI proposed Vitest + jsdom and the minimal config — saved reading docs.
- **Test generation:** I asked it to test `MFEBridge`, but steered hard on *what*: "test
  the runtime `Set` guard and origin/target filtering; don't test rendering". Generic
  "write tests for this file" gives you shallow getter/setter tests; the value is in
  naming the risks (the [1.1](../1-foundations/1.1-prompt-and-context-engineering.md) lesson).
- **Edge cases:** AI is good at enumerating cases I'd forget — disallowed origin, target
  mismatch, post-unsubscribe delivery. I kept the ones that map to real behavior and
  dropped the trivial ones.
- **Fixing the lint failure:** the generated mocks used `() => {}`, which the project's
  `no-empty-function` rule rejects. The verification step (`/verify`) caught it; fix was
  `() => undefined`. Good example of *tests must pass the same gate as code*.

> 💡 Takeaway: AI testing is only as good as the risks you name. "Write tests" → coverage
> theatre. "Test this specific failure mode" → a regression guard that earns its keep.

## Honest limitations

- Unit tests here cover **logic**, not the rendered game. Visual/interaction testing
  (does the card actually flip on screen?) would need Playwright + a canvas strategy —
  out of scope for this block, and AI-generated visual tests are still flaky.
- AI will happily generate tests that pass but assert nothing meaningful. Reviewing each
  test for "would this actually fail if the behavior broke?" is non-negotiable.

## Checklist for myself

- [ ] Is the test stack appropriate (Vitest+jsdom for DOM logic, not canvas)?
- [ ] Do tests target the **risky** units, not just easy ones?
- [ ] Is there a test that fails if the project's known bug returns?
- [ ] Do tests pass the same lint/type gate as production code?
- [ ] Did I review each test for a meaningful assertion (not coverage theatre)?

## Result

`memory-cards` went from no tests to a green Vitest suite (8 tests) wired into CI,
with the recurring protocol bug now guarded by an explicit test. AI did the heavy
lifting on setup and case enumeration; the judgment about *what's worth testing*
stayed with me.
