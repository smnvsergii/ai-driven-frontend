# 2.1 AI Code Review

**🌐 Language:** **English** · [Українська](./2.1-ai-code-review.ua.md)

**Block:** Daily Workflow
**Practice project:** [`phaser-memory-cards-MFE`](https://github.com/smnvsergii/phaser-memory-cards-MFE) — Phaser 3 + Vite + TypeScript memory game that also runs as a microfrontend.

> Goal of this note: make AI review useful by pointing it at what tooling **can't** catch — project-specific invariants — instead of letting it re-flag formatting that CI already enforces.

---

## The core insight

AI review fails when it duplicates the linter: it floods you with "add a semicolon" noise and misses the bug that actually matters. This project already runs `lint → format:check → typecheck → build` in [`ci.yml`](https://github.com/smnvsergii/phaser-memory-cards-MFE/blob/main/.github/workflows/ci.yml) and a pre-commit hook. So mechanical quality is **already solved**.

AI review earns its place only by checking the things a linter is blind to:

| A linter catches | Only a reviewer (human or AI) catches |
|------------------|----------------------------------------|
| Formatting, unused vars, `any` | "This flip animation touches `scaleY` — cards will jump" |
| Type errors | "You added a protocol command but forgot the runtime `Set`" |
| Build failures | "This hardcodes a size that belongs in `config.ts`" |

> 💡 Takeaway: AI review is only worth running if it covers what CI cannot. Define that scope explicitly, or you get noise.

## What I added to `memory-cards`

Three artifacts that turn "review my code" into a repeatable, project-aware process.

### 1. The rubric — `.github/CODE_REVIEW_CHECKLIST.md`

A curated checklist of this project's real risks, grouped by area:

- **Architecture & rendering** — canvas not DOM, no magic numbers (use `config.ts`), scene responsibilities.
- **Animation invariants** — the card flip animates `scaleX` only, never `scaleY` (the exact bug from [1.1](../1-foundations/1.1-prompt-and-context-engineering.md)).
- **MFE protocol** — type union ↔ runtime `SHELL_COMMAND_TYPES` `Set` sync, envelope pattern, shell mirror, origin policy.
- **General** — no needless deps, focused diff, `console` discipline, standalone mode preserved.

The first line of the file scopes it: *cover only what CI cannot*. This is the single source of truth for review criteria — humans and the AI use the same list.

### 2. The command — `.claude/commands/review.md`

A `/review` slash command (building on [1.3](../1-foundations/1.3-custom-skills-and-commands.md)) that:

- pulls the diff (`git diff origin/main...HEAD` + uncommitted changes),
- reads the checklist and reviews **strictly against it**,
- is explicitly told **not** to comment on formatting/lint/types,
- groups findings as **Blocker / Should-fix / Nit** with file:line references,
- is told to say "this is clean" rather than invent issues.

```
allowed-tools: Read, Bash(git diff:*), Bash(git status)
```

Note the scoped tools: the command can read and diff, but can't write or push — review should never mutate code.

### 3. The workflow hook — `.github/pull_request_template.md`

A PR template with a self-review checklist that mirrors the rubric and nudges:
*"run the `/review` AI command before pushing"*. This puts AI review **before** the PR, not as an afterthought.

## How it fits the pipeline

```
local change
  └─ /review            ← AI review against project rubric (catches design bugs)
  └─ /verify            ← typecheck + lint + build (from 1.3)
git push / open PR
  └─ pull_request_template.md   ← human self-review, same rubric
  └─ ci.yml             ← mechanical gate (lint/format/typecheck/build)
```

AI review sits at the *start*, where fixing a design mistake is cheapest. CI stays the mechanical gate. The two don't overlap — by design.

## Honest limitations

- AI review **complements**, not replaces, human review for anything security- or architecture-critical. It's a first pass that catches the obvious-in-hindsight stuff.
- It can still miss subtle logic bugs and hallucinate confident-but-wrong findings — hence the "don't invent issues" instruction and the Blocker/Should-fix/Nit grading, so I can triage fast.

## Checklist for myself

- [ ] Does the AI review scope explicitly exclude what CI already checks?
- [ ] Is there a single rubric used by both humans and the AI?
- [ ] Does the rubric encode *this project's* real invariants, not generic advice?
- [ ] Is the review command read-only (no write/push tools)?
- [ ] Is AI review positioned before the PR, where fixes are cheapest?

## Result

Review feedback went from "semicolon noise" to "you forgot the runtime `Set`
again". Pointing the AI at a project-specific rubric — and forbidding it from
repeating CI — is what made it actually useful.
