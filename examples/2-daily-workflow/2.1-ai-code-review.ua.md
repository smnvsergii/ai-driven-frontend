# 2.1 AI Code Review

**🌐 Мова:** [English](./2.1-ai-code-review.md) · **Українська**

**Блок:** Daily Workflow
**Практичний проєкт:** [`phaser-memory-cards-MFE`](https://github.com/smnvsergii/phaser-memory-cards-MFE) — гра «знайди пару» на Phaser 3 + Vite + TypeScript, яка також працює як мікрофронтенд.

> Мета цієї нотатки: зробити AI-рев'ю корисним, націливши його на те, що інструменти **не можуть** зловити — специфічні для проєкту інваріанти — замість того щоб дати йому повторювати форматування, яке CI вже забезпечує.

---

## Ключова ідея

AI-рев'ю провалюється, коли дублює лінтер: воно заливає тебе шумом «додай крапку з комою» й пропускає баг, який реально важить. Цей проєкт уже ганяє `lint → format:check → typecheck → build` у [`ci.yml`](https://github.com/smnvsergii/phaser-memory-cards-MFE/blob/main/.github/workflows/ci.yml) і pre-commit хук. Тож механічна якість **уже вирішена**.

AI-рев'ю заслуговує своє місце лише тоді, коли перевіряє те, до чого лінтер сліпий:

| Що ловить лінтер | Що ловить лише рев'юер (людина або AI) |
|------------------|------------------------------------------|
| Форматування, невикористані змінні, `any` | «Ця анімація перевороту чіпає `scaleY` — картки стрибатимуть» |
| Помилки типів | «Ти додав команду протоколу, але забув рантайм-`Set`» |
| Провали збірки | «Це хардкодить розмір, якому місце в `config.ts`» |

> 💡 Висновок: AI-рев'ю варте запуску лише тоді, коли покриває те, що CI не може. Визнач цю область явно — інакше отримаєш шум.

## Що я додав у `memory-cards`

Три артефакти, що перетворюють «зроби рев'ю мого коду» на повторюваний, обізнаний про проєкт процес.

### 1. Рубрика — `.github/CODE_REVIEW_CHECKLIST.md`

Підібраний чекліст реальних ризиків цього проєкту, згрупований за областями:

- **Architecture & rendering** — canvas, а не DOM, без магічних чисел (використовуй `config.ts`), відповідальності сцен.
- **Animation invariants** — переворот картки анімує лише `scaleX`, ніколи `scaleY` (той самий баг із [1.1](../1-foundations/1.1-prompt-and-context-engineering.ua.md)).
- **MFE protocol** — синхрон type union ↔ рантайм-`SHELL_COMMAND_TYPES` `Set`, патерн envelope, дзеркало в shell, origin policy.
- **General** — без зайвих залежностей, сфокусований дифф, дисципліна `console`, збережений standalone-режим.

Перший рядок файлу задає область: *покривай лише те, що CI не може*. Це єдине джерело істини для критеріїв рев'ю — люди й AI використовують той самий список.

### 2. Команда — `.claude/commands/review.md`

Slash-команда `/review` (продовження [1.3](../1-foundations/1.3-custom-skills-and-commands.ua.md)), яка:

- витягує дифф (`git diff origin/main...HEAD` + незакомічені зміни),
- читає чекліст і робить рев'ю **строго за ним**,
- має явну вказівку **не** коментувати форматування/лінт/типи,
- групує знахідки як **Blocker / Should-fix / Nit** з посиланнями file:line,
- має вказівку казати «тут чисто», а не вигадувати проблеми.

```
allowed-tools: Read, Bash(git diff:*), Bash(git status)
```

Зверни увагу на обмежені інструменти: команда може читати й робити дифф, але не може писати чи пушити — рев'ю ніколи не має змінювати код.

### 3. Гачок процесу — `.github/pull_request_template.md`

PR-шаблон із чеклістом самоперевірки, що дзеркалить рубрику й підштовхує:
*«запусти AI-команду `/review` перед пушем»*. Це ставить AI-рев'ю **перед** PR, а не постфактум.

## Як це вписується в пайплайн

```
локальна зміна
  └─ /review            ← AI-рев'ю за рубрикою проєкту (ловить дизайн-баги)
  └─ /verify            ← typecheck + lint + build (з 1.3)
git push / відкриття PR
  └─ pull_request_template.md   ← самоперевірка людини, та сама рубрика
  └─ ci.yml             ← механічний гейт (lint/format/typecheck/build)
```

AI-рев'ю стоїть на *початку*, де виправити дизайн-помилку найдешевше. CI лишається механічним гейтом. Вони не перетинаються — навмисно.

## Чесні обмеження

- AI-рев'ю **доповнює**, а не замінює людське рев'ю для всього критичного щодо безпеки чи архітектури. Це перший прохід, що ловить очевидне-заднім-числом.
- Воно все ще може пропустити тонкі логічні баги й галюцинувати впевнені-але-хибні знахідки — звідси інструкція «не вигадуй проблеми» й градація Blocker/Should-fix/Nit, щоб я швидко тріажив.

## Чекліст для себе

- [ ] Чи область AI-рев'ю явно виключає те, що CI вже перевіряє?
- [ ] Чи є єдина рубрика, яку використовують і люди, і AI?
- [ ] Чи кодує рубрика реальні інваріанти *цього проєкту*, а не загальні поради?
- [ ] Чи команда рев'ю «лише читання» (без інструментів запису/пушу)?
- [ ] Чи стоїть AI-рев'ю перед PR, де виправлення найдешевші?

## Результат

Фідбек рев'ю перетворився з «шуму про крапки з комою» на «ти знову забув рантайм-`Set`».
Націлювання AI на специфічну для проєкту рубрику — і заборона повторювати CI — це те,
що зробило його реально корисним.
