## The failure mode I'm fixing

Ask an AI "why is this broken?" and it often jumps straight to a plausible-looking
patch. Sometimes it works; often it fixes a *symptom* and the real bug resurfaces
elsewhere. From the rules section in [1.2](../1-foundations/1.2-ai-rules-and-custom-instructions.md):
*if an approach fails, diagnose the root cause instead of patching incrementally.*
Debugging is where that matters most.

The fix is to impose a **method** and to feed the AI this project's **known failure
modes**, so it checks the usual suspects before theorizing.

## The method

```
1. Restate the symptom   (observed vs expected; get a repro)
2. Hypothesize           (2вАУ3 ranked guesses, before reading code)
3. Locate with evidence  (confirm/kill each from the source)
4. Minimal root-cause fix(not a symptom patch)
5. Guard the regression  (assert / type / documented invariant)
```

Steps 2 and 3 are the ones humans (and eager AIs) skip. Writing hypotheses *before*
opening the code stops you from pattern-matching to the first thing you see.

## What I added to `memory-cards`

A `.claude/commands/debug.md` command (`/debug <symptom>`) that encodes the method
above **and** a catalogue of this project's real failure modes вАФ so the AI checks
them first:

| Symptom | Usual root cause |
|---------|------------------|
| Shell command ignored | Type added to `ShellCommand` union but **not** to the runtime `SHELL_COMMAND_TYPES` `Set`; or origin/`target` mismatch |
| No sound | Browser autoplay policy вАФ audio context needs a user gesture |
| Cards jump size mid-flip | A tween touched `scaleY` (must be `scaleX`-only) |
| Layout broken after resize | Relayout not driven by the scene `resize` event / config values |
| Works embedded but not standalone | `window.parent === window` branch in `MFEBridge` |

Tool scope is read-only-ish: `Read`, `git diff`, `git log`, `typecheck` вАФ it can
investigate but not rewrite the codebase or commit.

## Worked example: the silently-dropped command

Real bug class from this project вАФ perfect to show the method.

**1. Symptom:** the shell sends `pause`, the game keeps running. No error in the console.

**2. Hypotheses (before reading code):**
- (a) The `pause` handler isn't registered.
- (b) The message is filtered out before reaching handlers.
- (c) Origin/target mismatch drops the message.

**3. Locate with evidence:** open `bridge.ts`. The `handleMessage` guard calls
`isShellCommandType(data.type)` against the `SHELL_COMMAND_TYPES` `Set`. The `pause`
type exists in the `ShellCommand` union (so it type-checks) but is **missing from
the `Set`** вЖТ the guard returns `false` вЖТ message dropped silently. Hypothesis (b)
confirmed; (a) and (c) killed.

**4. Root-cause fix:** add `'pause'` to `SHELL_COMMAND_TYPES`. Not "add a special
case for pause" вАФ the root cause is the union/`Set` desync.

**5. Guard:** this is exactly why [1.3](../1-foundations/1.3-custom-skills-and-commands.md)'s
`/add-mfe-message` command exists and why the [2.1](./2.1-ai-code-review.md) review
checklist flags it. The debugging session fed back into the tooling so the bug can't
recur quietly.

> рЯТ° Takeaway: a good debug ends by making the bug *un-reintroducible* вАФ a guard, a
> command, a checklist item вАФ not just a one-off fix.

## Honest limitations

- AI is strong at the *mechanical* parts (reading stack traces, scanning for the
  desync) and weak at bugs needing runtime state it can't see. For those, it should
  ask for logs / a repro rather than theorize вАФ the command tells it to.
- It can state a wrong root cause confidently. The "confirm with evidence from the
  source" step is what catches that; don't accept a fix without the evidence.

## Checklist for myself

- [ ] Did the AI restate the symptom and get a repro before patching?
- [ ] Were hypotheses written *before* reading code?
- [ ] Is each hypothesis confirmed/killed with evidence from the source?
- [ ] Does the fix address the root cause, not a symptom?
- [ ] Did I add a guard so the bug can't return silently?

## Result

Debugging stopped being "try a patch, rerun, repeat". The `/debug` command's known
failure modes mean the AI usually finds the real cause on the first pass вАФ and the
fix loops back into a command or checklist so it stays fixed.
